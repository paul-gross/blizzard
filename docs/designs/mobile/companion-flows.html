Flow decisions

The inverse of the restaurant glance: a time window, not a moment. Outcomes first (shipped / got stuck / spent), then the night as a chronological tape for anyone who wants the story. Delivered as an 07:30 push; the screen is just the push, expanded.

"Got stuck" rows deep-link straight into the ask/needs-human flows from sheet 1 — the briefing is also the morning's to-do list.

Everything here is derived from the event log over a window — no new backend state, one new windowed rollup query.

Reuses event-log, chunks.query, fleet-spend, cost-format, status pills. New: windowed digest query, digest shell, scheduled push.

Flow decisions

Alarm, evidence, verb — in that order. The banner states the trigger; the timeline is the evidence (three identical redo laps with the same rationale read as a loop at a glance); the verb is right below it.

Pause is destructive-adjacent, so it gets the design language's first slide-to-confirm — a tap can misfire in a pocket, a 200px drag can't. Escalation (pause the whole runner) and de-escalation (snooze) flank it as plain buttons.

Pause, not stop: reversible from the same screen, and the desk-side debugging happens later. The alarm threshold ($/hour and redo-count per chunk) is hub config, not a mobile setting.

Reuses chunk-timeline rows + verdict colors, pause/resume endpoints, spend from ChunkUsageTotalView. New: alarm rule + push, alarm banner, slide-to-confirm control.

Flow decisions

The ask flow's big sibling: same shape (context → decision → resume), but the context is a shippable change. The screen argues for trust in three beats — the reviewer's verdict with its rationale, CI, then the diffstat as shape (mostly the SSE lib, mostly adds, specs grew) rather than a diff to read at 13px.

Approve is green and plain-tap — it's the expected outcome of a passed review, not a destructive act. Send-back requires a note (the agent needs a reason to act on); the GitHub link is the escape hatch for the change that needs desk-level scrutiny.

Lives in the Asks tab: an awaiting-delivery chunk is a question — "ship it?"

Reuses chunk-awaiting-human resume endpoint, chunk-facts, pm pointer → PR link. New: delivery-card shell, diffstat read (forge API via hub), diffstat bars.

Flow decisions

Ops for the machinery, not the work — a different audience from every other screen, so it lives under the Fleet tab. The triage order is fixed: what's wrong (heartbeat facts, stated as evidence, not adjectives), what's at risk (leases with ages), then the verbs.

The verbs are deliberately hub-side only — pause assignment, release leases, subscribe to recovery. Restarting the daemon itself is a desk job (systemd on the host); the phone's job is to stop the bleeding and make the fleet route around the dead runner.

"Notify me when it heartbeats again" closes the loop: on-call means you can put the phone away.

Reuses health + runners queries, pause endpoint, compactRef. New: runner-detail shell, lease release verb, heartbeat-recovery push.

Across both sheets, the mobile app is now eight flows: glance, answer, show-off (sheet 1) + digest, feed, protect, approve, repair (this sheet). The write surface stays tiny and high-leverage — answer an ask, ingest, pause, approve delivery, release a lease — every one an existing hub endpoint. Net-new plumbing introduced by this sheet: the windowed digest query, the spend/redo alarm rule + push channel, a forge diffstat read, and the heartbeat-recovery notification. The push channel is shared by four of the five flows — it's the first thing to build.
